## What is ResolveAI?

Multi-channel AI customer-ops platform for Pakistani fintech/e-commerce.
Automates customer support — orders, refunds, accounts, tickets, escalations —
in Roman Urdu, Urdu script, and English.

**Stack:** FastAPI + LangGraph + pgvector + Redis + OpenAI gpt-4o-mini
**Infra:** Docker Compose (db, redis, api, worker, langfuse, grafana, prometheus)

## Agent Graph — Node Order

```
classify_intent → redact_pii → retrieve → plan_tools → execute_tools
→ compose_response → critique → (retry or escalate) → send_reply
```

### Intent Labels
- `order_status` — SPECIFIC order (ORD-XXX present) — status, tracking, cancellation, confirmation
- `refund_request` — explicitly CREATING a new refund ("I want a refund", "paise wapas chahiye")
- `account_inquiry` — account balance, transactions, frozen/active status
- `technical_support` — bugs, crashes, login issues, wrong charges/billing, payment failures
- `general_inquiry` — policy/how-to questions (delivery times, return policy, payment methods)
- `escalate_human` — asking for human/manager/supervisor
- `abuse` — threats or inappropriate content
- `session_end` — goodbye, shukriya, bye

### Critical Classification Rules (in classify_intent.yaml)
- If message has ORD-XXX → always `order_status`, never `account_inquiry`
- "How long does delivery take?" / "How do I track orders?" = `general_inquiry` (policy, not a specific order)
- "Wrong amount charged" / "incorrect billing" = `technical_support`
- "Why is MY account frozen?" (میرا/mera + کیوں/kyun) = `account_inquiry` (personal, not policy)
- Checking if past refund arrived ("kya paise aaye?") = `account_inquiry`

### Tools
- `get_order_status(order_id)` — live order from mock CRM
- `get_account_balance(user_id)` — balance + status (frozen/active)
- `get_recent_transactions(user_id)` — last N transactions
- `create_refund_request(order_id, reason, user_id)` — submit refund
- `create_support_ticket(user_id, summary, priority)` — create ticket

### Critical Tool Rules (in plan_tools.yaml)
- `technical_support` intent + customer REPORTING a problem → ALWAYS call `create_support_ticket`
- `account_inquiry` with "my account" / "mera" / "میرا" → ALWAYS call `get_account_balance` even if they ask WHY
- `refund_request` + explicit ask + order ID present → call `create_refund_request`
- "How do I get a refund?" (informational) → NO tool, explain process and ask for order ID

## Phase 12 — Eval Harness

### Files
- `data/eval/golden_set.jsonl` — 100 JSONL eval cases
- `tests/eval/run_eval.py` — main runner (exit code 1 if below thresholds)
- `tests/eval/llm_judge.py` — LLM-as-judge using gpt-4o-mini
- `tests/eval/metrics.py` — deterministic assertions + threshold enforcement
- `reports/eval_report.html` — last HTML report (generated by eval)

### Thresholds
- pass_rate ≥ 85%
- avg_groundedness ≥ 0.75
- avg_helpfulness ≥ 0.75
- avg_policy_score ≥ 0.75

### Run Commands (inside Docker)
```bash
# Full 100-case eval with HTML report
docker compose exec -T api python -m tests.eval.run_eval --html reports/eval_report.html

# Smoke test — first 5 cases
docker compose exec -T api python -m tests.eval.run_eval --limit 5 --no-db

# Only specific cases (e.g. previously failing ones)
docker compose exec -T api python -m tests.eval.run_eval --no-db --cases "tkt_002,faq_007,acc_007"

# Skip LLM judge (faster, cheaper — no rubric scores)
docker compose exec -T api python -m tests.eval.run_eval --no-judge
```

### Eval Philosophy — IMPORTANT
**Do NOT use deterministic must_include/must_not_include on AI response wording.**
LLM judge (gpt-4o-mini) owns semantic correctness, grounding, helpfulness.
Deterministic checks are ONLY for hard identifiers:
- ✅ OK: order IDs (ORD-XXX), tracking numbers (TRK-PKG-XXX), "ticket", "agent", tool calls
- ❌ NEVER: status words (dispatched/delivered), currency format (PKR vs rupees), exact Urdu phrases, escalation words (escalat/connect)

### Phase 13 — CI/CD

**Files**
- `.github/workflows/ci.yml` — 3 jobs: lint → test → eval-smoke (main only)
- `.github/workflows/deploy.yml` — triggered by CI success; builds prod image → pushes to GHCR → SSH restarts stack

**ci.yml jobs**
- `lint` — ruff + black (host Python 3.11, no services)
- `test` — pgvector + redis service containers; runs migrations → seed → `pytest tests/unit tests/integration` with coverage
- `eval` (main/master only) — same services + ingest KB → `run_eval --limit 10`; uploads HTML report as artifact

**deploy.yml**
- Triggered via `workflow_run` on CI success (main/master)
- Builds `Dockerfile target: production`, pushes to `ghcr.io/<owner>/<repo>:latest`
- SSH into server: pulls image, restarts `api` + `worker` via `docker-compose.prod.yml`, health-checks `/healthz`

**GitHub Secrets required**
| Secret | Used by |
|---|---|
| `OPENAI_API_KEY` | eval job (real LLM calls) |
| `DEPLOY_HOST` | deploy SSH target |
| `DEPLOY_USER` | deploy SSH user |
| `DEPLOY_SSH_KEY` | deploy SSH private key |
| `GITHUB_TOKEN` | automatic — GHCR push + pull |

**Server pre-requisites**
- `~/resolveai/docker-compose.prod.yml` and `~/resolveai/.env` present
- `RESOLVEAI_IMAGE` env var consumed by compose prod file to pin the pulled image
